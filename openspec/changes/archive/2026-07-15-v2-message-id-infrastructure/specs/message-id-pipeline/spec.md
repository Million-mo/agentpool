## ADDED Requirements

### Requirement: Single source of truth for message_id across the event pipeline

The system SHALL maintain a single `message_id` per logical message that flows from generation (native turn or ACP inbound) through events, CommChannel feedback, and protocol conversion. No component in the pipeline SHALL generate an independent `message_id` for a message that already has one.

- `NativeTurn._message_id` SHALL be the origin for native agent messages. It SHALL be propagated to `PartStartEvent.message_id`, `PartDeltaEvent.message_id`, `ToolCallCompleteEvent.message_id`, and the final `ChatMessage.message_id`.
- `ACPMessageAccumulator` SHALL preserve the incoming `message_id` from external ACP agents' `AgentMessageChunk`, `UserMessageChunk`, and `AgentThoughtChunk` session updates. When the incoming `message_id` is `None` or empty, it SHALL fall back to `str(uuid4())`.
- `ACPEventConverter` SHALL read `message_id` from streaming events (`PartStartEvent`, `PartDeltaEvent`) instead of maintaining an independent `_current_message_id`. For events without a `message_id` (error events, compaction events), it SHALL generate a one-off UUID inline.
- The OpenCode server event processor SHALL read `message_id` from streaming events instead of generating `assistant_msg_id` independently.

#### Scenario: Native agent message_id flows end-to-end

- **WHEN** a native agent turn starts and generates `_message_id = uuid4().hex`
- **THEN** `PartStartEvent` SHALL carry that `message_id`
- **AND** all subsequent `PartDeltaEvent`s for the same message SHALL carry the same `message_id`
- **AND** `ACPEventConverter` SHALL use that `message_id` on `AgentMessageChunk` notifications
- **AND** the final `ChatMessage.message_id` SHALL match the event `message_id`

#### Scenario: External ACP agent message_id preserved

- **WHEN** an external ACP agent (e.g., Goose) sends `AgentMessageChunk` with `message_id="msg_abc123"`
- **THEN** `ACPMessageAccumulator.process()` SHALL read `update.message_id`
- **AND** `_finalize_current_message()` SHALL use `"msg_abc123"` as the `ChatMessage.message_id`
- **AND** no fresh UUID SHALL be generated when a valid `message_id` is present

#### Scenario: External ACP agent without message_id

- **WHEN** an external ACP agent sends `AgentMessageChunk` with `message_id=None`
- **THEN** `ACPMessageAccumulator._finalize_current_message()` SHALL generate `str(uuid4())` as fallback
- **AND** the generated ID SHALL be used for `ChatMessage.message_id`

#### Scenario: ACPEventConverter reads from events

- **WHEN** `ACPEventConverter` processes a `PartStartEvent` with `message_id="msg_001"`
- **THEN** the resulting `AgentMessageChunk` SHALL carry `message_id="msg_001"`
- **AND** no independent UUID SHALL be generated by the converter

#### Scenario: Error event without message_id generates one-off UUID

- **WHEN** `ACPEventConverter` processes a `RunErrorEvent` (which has no `message_id` field)
- **THEN** the converter SHALL generate a one-off `str(uuid4())` for the resulting `AgentMessageChunk`
- **AND** this UUID SHALL NOT be stored as a persistent `_current_message_id`

### Requirement: PartStartEvent and PartDeltaEvent carry message_id

`PartStartEvent` and `PartDeltaEvent` SHALL gain a `message_id: str` field with default `""`. All chunks belonging to the same logical message SHALL share the same `message_id`. An empty string means "not set" — protocol converters SHALL treat `""` as absent for v1 optional semantics.

- `NativeTurn` SHALL set `message_id` on `PartStartEvent` using `self._message_id`
- `NativeTurn` SHALL propagate the same `message_id` to all `PartDeltaEvent`s for that message
- `ACPTurn` SHALL set `message_id` from the incoming ACP session update's `message_id` field, or generate a new UUID if absent
- The `message_id` on `PartStartEvent` SHALL NOT change during the lifetime of a single message

#### Scenario: PartStartEvent carries message_id

- **WHEN** a native agent starts streaming a new text part
- **THEN** `PartStartEvent.message_id` SHALL be set to the turn's `_message_id`
- **AND** it SHALL be a non-empty string

#### Scenario: PartDeltaEvent inherits message_id

- **WHEN** a `PartDeltaEvent` is emitted as a continuation of a `PartStartEvent`
- **THEN** `PartDeltaEvent.message_id` SHALL match `PartStartEvent.message_id`

#### Scenario: Empty message_id treated as absent

- **WHEN** `PartStartEvent.message_id` is `""`
- **THEN** `ACPEventConverter` SHALL generate a one-off UUID for the ACP notification
- **AND** no assertion or crash SHALL occur

#### Scenario: Multiple PartStartEvents in a single native turn share message_id

- **WHEN** a native agent emits multiple `PartStartEvent`s within a single turn (e.g., text → tool call → more text)
- **THEN** all `PartStartEvent`s SHALL carry the same `message_id` (the turn's `_message_id`)
- **AND** all `PartDeltaEvent`s between them SHALL also carry that same `message_id`

#### Scenario: External ACP agent sends multiple chunks with different message_ids

- **WHEN** an external ACP agent sends `AgentMessageChunk(message_id="msg_001", text="Hello")` followed by `AgentMessageChunk(message_id="msg_002", text="World")` without a role switch
- **THEN** `ACPMessageAccumulator.process()` SHALL detect the `message_id` change from `"msg_001"` to `"msg_002"`
- **AND** SHALL trigger `_finalize_current_message()` for the first message (preserving `message_id="msg_001"`)
- **AND** SHALL start accumulating the second message with `message_id="msg_002"`
- **AND** both messages SHALL appear in the final `ChatMessage` history with their respective IDs

#### Scenario: External ACP agent sends chunks with same message_id

- **WHEN** an external ACP agent sends multiple `AgentMessageChunk` notifications with the same `message_id="msg_001"`
- **THEN** `ACPMessageAccumulator` SHALL accumulate them into a single message
- **AND** SHALL NOT trigger `_finalize_current_message()` between chunks with the same `message_id`

#### Scenario: content_blocks carried through pipeline without stringification

- **WHEN** `receive_request()` receives `content` as a `list` (e.g. `[{"type": "image", ...}, "caption"]`)
- **THEN** the list SHALL be preserved as-is in `Feedback.content_blocks` (not stringified)
- **AND** `Feedback.content` SHALL be set to a plain-text fallback (e.g. empty string or joined text parts)
- **AND** `steer()` SHALL call `agent_run.enqueue(*content_blocks)` (unpacking the list) for native agents
- **AND** `steer()` SHALL call `agent_run.enqueue(content)` when `content_blocks` is `None` (backward-compatible text path)

#### Scenario: Protocol-specific ContentBlock mapping is deferred

- **WHEN** ACP `ContentBlock[]` or OpenCode `parts` arrive at the protocol boundary
- **THEN** conversion to PydanticAI `UserContent` (ImageUrl, BinaryContent, TextContent) SHALL be performed by the v2 protocol adapter (future work)
- **AND** the internal pipeline treats `content_blocks` as opaque `list[Any]` — no type enforcement
- **AND** no assertion or error SHALL occur when `content_blocks` contains protocol-specific dict shapes

### Requirement: OpenCode delivery mode maps to receive_request priority

The OpenCode protocol's `delivery: "steer" | "queue"` field SHALL map to `SessionController.receive_request()` priority routing. OpenCode route handlers SHALL pass `delivery` from `MessageRequest` to `receive_request()` instead of hardcoding `priority="when_idle"`.

- `delivery: "steer"` SHALL map to `priority: "asap"` (mid-turn injection)
- `delivery: "queue"` SHALL map to `priority: "when_idle"` (next-turn queue)
- When `delivery` is not provided, the default SHALL be `"queue"` (matching OpenCode's default)

#### Scenario: OpenCode client sends steer delivery

- **WHEN** an OpenCode client sends `POST /message` with `delivery: "steer"` on a busy session
- **THEN** `receive_request()` SHALL be called with `priority="asap"`
- **AND** the message SHALL be injected mid-turn via `RunHandle.steer()`

#### Scenario: OpenCode client sends queue delivery

- **WHEN** an OpenCode client sends `POST /message` with `delivery: "queue"` on a busy session
- **THEN** `receive_request()` SHALL be called with `priority="when_idle"`
- **AND** the message SHALL be queued for the next turn via `RunHandle.followup()`

#### Scenario: OpenCode client omits delivery

- **WHEN** an OpenCode client sends `POST /message` without `delivery` field
- **THEN** `receive_request()` SHALL be called with `priority="when_idle"` (default queue behavior)

### Requirement: OpenCode server resolves dual assistant_msg_id

The OpenCode server SHALL use a single `assistant_msg_id` per turn, sourced from the event pipeline. The REST path (`message_routes.py`) generates the canonical ID using `identifier.ascending("message", request.message_id)`, passes it to `receive_request(message_id=...)`, and the EventBus consumer (`session_pool_integration.py`) reads `message_id` from `PartStartEvent` instead of generating its own.

- The REST path SHALL generate `assistant_msg_id` using `identifier.ascending("message", request.message_id)` and pass it to `receive_request(message_id=assistant_msg_id)`
- The EventBus consumer SHALL NOT generate its own `assistant_msg_id` in `_before_consumer_loop()` — it SHALL read `message_id` from `PartStartEvent`
- All parts (text, tools, reasoning, step-start/finish) SHALL link to the same `message_id`
- The `identifier.ascending()` format (`msg_*` prefix) is opaque to the internal pipeline and treated identically to UUID format

#### Scenario: Single assistant_msg_id per turn via OpenCode

- **WHEN** an OpenCode client sends a message and the agent streams a response
- **THEN** all parts (text, tools, reasoning, step-start/finish) SHALL carry the same `message_id`
- **AND** the `message_id` SHALL be the one generated by the REST path and propagated through events
- **AND** the EventBus consumer SHALL NOT generate a separate `assistant_msg_id`

#### Scenario: OpenCode client-provided message_id propagates

- **WHEN** an OpenCode client sends `POST /message` with `message_id: "msg_custom_001"`
- **THEN** `identifier.ascending("message", "msg_custom_001")` SHALL return `"msg_custom_001"`
- **AND** this ID SHALL propagate to `receive_request(message_id="msg_custom_001")`
- **AND** all events and parts SHALL carry `message_id="msg_custom_001"`
