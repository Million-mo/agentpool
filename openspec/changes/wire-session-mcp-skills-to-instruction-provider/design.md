## Context

AgentPool has two layers of MCP server management:

1. **Pool-level MCP servers** — defined in YAML config under `mcp_servers:`, managed by `MCPManager`, stored in `self.mcp.providers`. These are registered in the pool's `_skill_provider` aggregator during `_setup_skills_provider()`.

2. **Session-level MCP servers** — created dynamically when an ACP client sends `mcp_servers` in `session/new` (MCP-over-ACP). `ACPSession.initialize_mcp_servers()` creates `MCPResourceProvider` instances, stores them in `self.session_mcp_providers`, and adds them to the session agent's tool chain. They are **not** registered in the pool's skill aggregator.

The `SkillsInstructionProvider` generates `<available-skills>` XML for agent prompts. It queries `self.skill_provider.get_skills()`, which is the pool-level `AggregatingResourceProvider`. Since session-level providers are absent from this aggregator, MCP-over-ACP skills never appear in agent prompts.

The `SkillURIResolver` similarly only knows about pool-level providers, so `load_skill` with bare skill names cannot resolve session-level MCP skills either.

### Current Architecture

```
Pool.__aenter__()
  → MCPManager.__aenter__()  # connects pool-level MCP servers
  → _setup_skills_provider()
      providers = [LocalResourceProvider, *pool.mcp.providers]
      self._skill_provider = AggregatingResourceProvider(providers)
      self._skill_resolver = SkillURIResolver(providers)
  → SkillsInstructionProvider.skill_provider = self._skill_provider

... later, when ACP client connects ...
ACPSession.initialize_mcp_servers()
  → for each mcp_server in request:
      provider = MCPResourceProvider(...)
      self.session_mcp_providers.append(provider)    # ← session-local
      session_agent.tools.add_provider(provider)     # ← tools work
      # provider NOT added to pool._skill_provider   # ← BUG: skills missing
```

## Goals / Non-Goals

**Goals:**
- Session-level MCP provider skills appear in `<available-skills>` XML generated by `SkillsInstructionProvider`
- `load_skill("skill-name")` resolves skills from session-level MCP providers
- `list_skills` includes session-level MCP skills
- No change to how pool-level MCP providers work
- Provider deregistration on session teardown to avoid stale references

**Non-Goals:**
- Changing how `MCPResourceProvider.get_skills()` discovers skills (already works correctly)
- Changing the ACP protocol itself
- Adding `list_skills`/`list_resources` to ACP protocol messages
- Refactoring the overall provider architecture

## Decisions

### Decision 1: Add a `register_skill_provider()` method to `AgentPool`

**Choice**: Add `AgentPool.register_skill_provider(provider: ResourceProvider) -> None` that appends to the aggregator and resolver.

**Rationale**: The pool already owns `_skill_provider` and `_skill_resolver`. A registration method keeps ownership clear and avoids exposing internal state to `ACPSession`.

**Alternatives considered**:
- *Direct mutation of `pool._skill_provider.providers`* — rejected: breaks encapsulation, requires `AggregatingResourceProvider` to support dynamic mutation
- *Having `SkillsInstructionProvider` query session-level providers separately* — rejected: requires the instruction provider to know about sessions, violates separation of concerns
- *Re-creating the aggregator on each session* — rejected: expensive, loses pool-level providers during rebuild

### Decision 2: Make `AggregatingResourceProvider` support dynamic provider addition

**Choice**: Add `add_provider(provider: ResourceProvider)` method that appends to `self.providers` and forwards `skills_changed` signal.

**Rationale**: `AggregatingResourceProvider` already wraps a list of providers. Making it mutable allows late registration without re-creation.

### Decision 3: Register providers in `ACPSession.initialize_mcp_servers()` via the pool

**Choice**: After creating each `MCPResourceProvider`, call `pool.register_skill_provider(provider)`. On session teardown, call `pool.unregister_skill_provider(provider)`.

**Rationale**: The session knows when providers are created and destroyed. The pool manages the aggregator. This is the natural handoff point.

### Decision 4: `SkillURIResolver` supports dynamic registration

**Choice**: Add `register_provider(name, provider)` and `unregister_provider(name)` methods.

**Rationale**: The resolver already has a `register_provider` method used during `_setup_skills_provider()`. Session-level providers just need to call the same method.

## Risks / Trade-offs

- **Risk**: Session-level provider remains registered after session teardown → stale skills in prompts
  - **Mitigation**: `unregister_skill_provider()` called in session cleanup; aggregator verifies provider is still connected before returning skills

- **Risk**: Skill name collisions between pool-level and session-level MCP providers
  - **Mitigation**: `AggregatingResourceProvider.get_skills()` already returns a flat list; `SkillURIResolver.resolve()` returns first match. Session-level providers are registered after pool-level, so pool-level skills take priority for bare-name resolution. For `skill://` URIs, the provider name disambiguates.

- **Risk**: Multiple concurrent sessions each registering their own providers → large provider list
  - **Mitigation**: Providers are unregistered on session teardown. The list only contains active session providers.

- **Trade-off**: `AggregatingResourceProvider.providers` changes from effectively-immutable to mutable
  - **Acceptable**: The provider list was already mutable in practice (it's a `list`). Adding explicit add/remove methods makes the contract clearer.
